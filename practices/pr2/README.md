# Сравнение RabbitMQ и Redis как брокеров сообщений

## Запуск

```bash
docker compose down -v && docker compose up --build 
```

---

## Описание 

Все три компонента запущены в одном `docker compose`:

- `rabbitmq` — `rabbitmq:3.13-management`, лимит 1 CPU + 512 МБ RAM.
- `redis` — `redis:7-alpine`, лимит 1 CPU + 512 МБ RAM.
- `runner` — Python-проги `producer.py`, `consumer.py`, `run_tests.py`.

Производитель и потребитель работают в двух параллельных потоках. Каждый тест использует уникальную очередь, что исключает влияние предыдущих прогонов.

Формат сообщения одинаков для обоих брокеров: 16-байтный заголовок плюс заполнение до нужного размера.

---

## Ошибка в первом прогоне

В самом первом прогоне (`results_run1`) сценарий `baseline [rabbitmq]` завершился с ошибкой:

```
pika.exceptions.AMQPConnectionError
ERROR: {'tag': 'baseline', 'broker': 'rabbitmq', ...}
```

**Причина:** проверка состояния `rabbitmq-diagnostics -q ping` отмечает контейнер как готовый раньше, чем AMQP на порту 5672 начинает принимать соединения.

**Исправление:** в `producer.py` и `consumer.py` добавлены функции `_connect_rabbit` и `_connect_redis` — циклы повторных попыток с интервалом 0.5 с и общим таймаутом 30 с. Начиная с прогона run2 ошибоки были устранены.

## Точки деградации

### RabbitMQ

При 1000 сообщений/с работает устойчиво: задержка 3–5 мс, очередь нулевая.

При 5000 сообщений/с средняя задержка скачет от 125 до 641 мс в зависимости от прогона. Потребитель начинает отставать — обрабатывает 3800–4200 из 5000 сообщений/с. Очередь по Management-API показывает 0, поскольку потребитель всё же успевает читать, просто медленно. Нагрузка выражается ростом задержки.

При 10 000 сообщений/с потребитель устойчиво держит лишь ~4400 сообщений/с. Средняя задержка стабильно превышает 1 секунду, p95 достигает 1820–1977 мс. 

При 20 000 сообщений/с картина не улучшается — максимум потребителя остаётся ~4800 сообщений/с, независимо от объёма подачи. Дополнительная нагрузка лишь растягивает время обработки, не увеличивая пропускную способность.

**Точка деградации RabbitMQ: около 4000–5000 сообщений/с** (1 производитель, 1 потребитель, 512 Б, 1 CPU).

### Redis

До 5000 сообщений/с задержка остаётся ниже 1 мс, очередь нулевая.

При 10 000 сообщений/с задержка вырастает до 0.8–14 мс. Очередь временами достигает 13–580 сообщений — Redis начинает буферизовать.

При 20 000 сообщений/с очередь резко растёт: 441–3731 сообщение. Средняя задержка 11–155 мс, потребитель держит 8700–10900 сообщений/с. Память Redis при этом прыгает с 1.13 МБ до 2.01–2.03 МБ.

**Точка деградации Redis: около 10 000–12 000 сообщений/с** при тех же вводных.

### Описание результатов прогонов 8 и 9

**Прогон 8 (6 CPU / 512 МБ)** показал резкое улучшение для RabbitMQ почти по всем сценариям с размером сообщения. Сценарий `size_100KB` стал самым показательным: задержка упала с 1472 мс до 4.16 мс — в 354 раза. При 5000 сообщений/с задержка снизилась с 318 мс до 18 мс. 

При высоких нагрузках (10k–20k) заметность меньше: потолок потребителя вырос с ~4400 до ~5200 сообщений/с (+18%). 

**Прогон 9 (1 CPU / 4 ГБ)** — результаты RabbitMQ, как и должно, остались близкими к базовым прогонам: baseline (558 мс), rate_10k (1413 мс). Небольшое улучшение на size_100KB (70 мс против 1472 мс в среднем) объясняется тем, что 4 ГБ позволили RabbitMQ разместить рабочие структуры в памяти без вытеснения.   
Redis в прогоне 9 показал лучшие результаты по задержке из всех прогонов: при rate_10k — 0.46 мс (против 5.9 мс в прогонах 4–6), при rate_20k — 1.87 мс при backlog max всего 55 сообщений.

## Немного о метриках памяти брокеров

Для RabbitMQ значения `broker_mem_avg_mb` близки к нулю (0.01–0.20 МБ). Это только память конкретной очереди согласно полю `memory` из Management API, а не полный объём RSS процесса. Полный RSS брокера составляет ~150–200 МБ, но он недоступен изнутри контейнера раннера без монтирования сокета Docker. Поэтому цифры для RabbitMQ отражают только накладные расходы на очередь, а не на брокер в целом.

Для Redis значения реальные: `INFO memory` возвращает `used_memory` всего процесса. При накоплении backlog (rate_20k) объём растёт с 1.1 МБ до 2.0 МБ — сообщения буквально лежат в памяти.

| параметр | теория | практика |

| Backlog в Redis | Растёт при перегрузке, виден через LLEN | Да. При 20k backlog: 441–3731 сообщение. Память при этом растёт до 2 МБ |
| Точка деградации RabbitMQ ниже | AMQP-overhead выше, значит порог ниже | Да. RabbitMQ ~5k msg/s, Redis ~12k msg/s — разрыв в 2.4 раза |

## Итоговые выводы

**Пропускная способность.** Redis превосходит RabbitMQ на всех сценариях. При базовой нагрузке (5000 сообщений/с, 1 КБ) Redis стабильно даёт ~4440 сообщений/с против ~3550 у RabbitMQ. При высоких нагрузках разрыв увеличивается: при 10 000 сообщений/с Redis держит ~8000 против ~4400 у RabbitMQ.

**Устойчивость к размеру сообщения.** Redis практически нечувствителен к росту размера — задержка меняется с 0.11 до 0.23 мс при переходе от 128 Б до 100 КБ. RabbitMQ деградирует нелинейно: при 100 КБ средняя задержка прыгает до 1500 мс, а в худшем прогоне достигает 2939 мс. Пропускная способность при этом падает с 1997 до 1109–1629 сообщений/с. Причина — AMQP-фрейминг.

**Точки деградации.** RabbitMQ начинает отставать уже при 5000 сообщений/с (задержка 100–640 мс). При 10 000 и выше потребитель устойчиво держит лишь ~4400 сообщений/с независимо от интенсивности подачи — это аппаратный потолок. Redis деградирует позже: первые признаки появляются при 10 000 сообщений/с (задержка до 14 мс, backlog до 580), полноценная потеря характера происходит при 20 000 сообщений/с (backlog до 3731, задержка 11–155 мс).

**Влияние ресурсов.** RabbitMQ хорошо масштабируется по CPU (прогон 8 при 6 ядрах): задержка на 100 КБ упала в 354 раза (с 1472 до 4.16 мс), на rate_5k — в 18 раз. Причина в архитектуре кролика: планировщик раскидывает процессы по ядрам, и при 1 CPU они конкурируют друг с другом. Память же почти не влияет на RabbitMQ — из прогона 9 (1 CPU / 4 ГБ) видня почти идентичные базовым 1 CPU / 512 МБ результаты. Redis иначе: он однопоточный, поэтому от дополнительных ядер не выигрывает, зато больше памяти полностью устраняет конкуренцию — задержка при rate_10k упала с 5.9 до 0.46 мс.

**Выбор инструмента.** Для простой очереди с высокой пропускной способностью и минимальной задержкой Redis лучше при любом уровне ресурсов. Однако он не является полноценным брокером: без настройки персистентности данные теряются при рестарте, нет подтверждений доставки, нет маршрутизации. RabbitMQ проигрывает по скорости на 1 CPU, но при достаточном числе ядер разрыв сокращается до 1.1–1.5 раз. При этом RabbitMQ даёт надёжность, персистентность, подтверждения на уровне протокола и гибкую маршрутизацию. 

**Итог**: для потоковой обработки задач с допустимой потерей — Redis; для систем, где потеря сообщения недопустима, — RabbitMQ с достаточным числом ядер.
